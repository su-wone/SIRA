- **NFR-S2:** 대시보드: HTTPS 필수, CORS 정책 설정 (허용 오리진 명시)

- **NFR-S4:** AI API 전송 시 코드 컨텍스트 최소화 (필요한 파일 경로/구조만)
- **NFR-S5:** RBAC 권한 위반 시 요청 거부 및 로깅

#### Scalability
- **NFR-SC1:** MVP: 단일 팀(2-10명), 단일 레포, 태스크 500개까지 안정 동작
- **NFR-SC2:** Growth: 태스크 5,000개, 멀티 프로젝트, 외부 팀 5개(총 2,500+ 태스크) 지원
- **NFR-SC3:** 인덱스 엔진: 증분 업데이트로 태스크 수 증가 시에도 성능 유지

- **NFR-I2:** Git 조작: `simple-git` 또는 동등한 라이브러리로 commit/push/pull 자동화
- **NFR-I3:** Claude API: Anthropic SDK (TypeScript) 사용, 모델 버전 설정 가능
- **NFR-I4:** Growth 단계에서 GitLab/Bitbucket provider 추가 가능한 어댑터 구조

#### Reliability
- **NFR-R1:** 양방향 싱크: 모든 충돌 상황에서 양쪽 변경사항이 최소 하나의 브랜치/커밋에 보존된다
- **NFR-R2:** 인덱스 불일치 시 자동 복구 (full rebuild fallback)
- **NFR-R3:** Git 통신 실패 시 재시도 (최대 3회, 지수 백오프) + 로컬 큐잉 (최대 100건)
- **NFR-R4:** `.sira/` 디렉토리 무결성 검증: config.yaml 손상 시 백업에서 복구, 스키마 검증

- 코드 컨텍스트가 Claude API로 전송됨 — 기업 고객 민감도 고려 필요

- API 키 팀 공유 구조의 키 유출 리스크 관리

#### Git 기반 아키텍처 제약
- 동시성 충돌: 다수 팀원 동시 .md 수정 시 Git 머지 충돌 → 필드 레벨 머지로 완화
- 대용량 레포: 태스크 수천 개 시 index.json 성능 저하 → 증분 인덱싱, 아카이브 전략
- 브랜치 전략: feature 브랜치 태스크 .md 파일의 대시보드 표시 기준 정의 필요

#### AI API 제약
- 비용: 자동 추론 빈번 호출 시 월 비용 급증 → 캐싱, 배치 처리, 로컬 LLM 옵션
- 레이턴시: 실시간 상태 추론은 매번 API 호출 불가 → 휴리스틱 + 주기적 AI 검증
- Rate Limit: 팀 동시 `sira ask` 사용 시 API 한도 초과 가능 → 큐잉, 요청 병합

#### 기술 제약
- TypeScript + Claude API 기반
- DB 없이 Git을 유일한 데이터 저장소로 사용
- Node.js 18+ 요구
- 크로스 플랫폼: macOS, Linux, Windows

### 주의 사항 (텍스트 불일치)

**FR1 텍스트 변경 (중요):**
- **PRD 원문:** "팀장은 **자연어로 백로그를 입력하여** 에픽/스토리/태스크를 **자동 생성**할 수 있다"
- **에픽 문서:** "팀장은 **BMAD 스타일의 구조화된 워크플로우를 통해 AI와 협업하여** 에픽/스토리/태스크를 **단계적으로 생성**할 수 있다"
- **영향:** PRD는 "자연어 → 자동 생성"을 강조하지만, 에픽은 "구조화된 워크플로우 → 단계적 생성"으로 접근 방식이 변경됨. 이는 사용자 경험에 의미 있는 차이이므로, PRD와 에픽 중 어느 쪽이 최종 의도인지 확인 필요.
- **추천:** PRD를 에픽 문서에 맞게 업데이트하거나, 에픽이 PRD 원문의 "자연어 자동 생성" 경험도 지원하는지 명확히 할 것.
