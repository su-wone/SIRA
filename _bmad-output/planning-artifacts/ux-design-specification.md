# UX Design Specification SIRA

**Author:** suwon
**Date:** 2026-03-11

---

## Executive Summary

### Project Vision

SIRA는 AI+Human 하이브리드 팀을 위한 작업 인텔리전스 플랫폼이다. .md 파일 + YAML frontmatter를 DB로, Git을 유일한 저장소로 사용하며, CLI(개발자)와 웹 대시보드(팀장/PM)를 핵심 인터페이스로 제공한다.

### Target Users

| 사용자 | 주요 목표 | 주 인터페이스 | 기술 수준 |
|--------|---------|-------------|---------|
| 팀장/PM (민수) | 팀 현황 파악, 태스크 할당/승인 | 웹 대시보드 | 중급 |
| 개발자 (지영) | 코드 작업 중 태스크 조회/변경 | CLI | 고급 |
| AI 에이전트 | 태스크 실행, 상태 자동 업데이트 | 프로그래매틱 | N/A |

### Key Design Challenges

1. **CLI↔Dashboard 양방향 싱크 UX** — 실시간 반영 + 충돌 시 직관적 선택 UI
2. **사람+AI 동등 표현** — 픽셀 오피스에서 AI도 캐릭터, 실시간 상태 표시
3. **정보 밀도 vs 가독성** — 통계+프로젝트+멤버+픽셀오피스를 과부하 없이 배치
4. **.md 렌더링↔편집 전환** — 마크다운 뷰/편집 모드 매끄러운 전환

### Design Opportunities

1. **픽셀 오피스 (게이미피케이션)** — 팀 활동을 RPG 스타일로 시각화, 강력한 차별점
2. **Zero-Input 상태 추론 UI** — AI 추론 결과를 "제안" 뱃지로, 한 클릭 승인/거부
3. **자연어 질의 바** — 대시보드에서 sira ask 스타일 자연어 검색 지원

### Design Reference (User-Provided Mockups)

- 팀 대시보드: 글로벌 통계 바 + 픽셀 오피스 (다크 테마)
- 프로젝트+멤버 뷰: 프로젝트 진행률 카드 + 멤버 활동 카드
- 태스크 테이블: 프로젝트별 그룹핑, 상태/우선순위 배지
- 간트 뷰: 프로젝트별 타임라인, 상태별 색상 코딩 (MVP 포함 확정)
- 태스크 상세: .md 렌더링 + 메타 배지 + 히스토리 테이블
- 색상 체계: 노란색=진행중, 보라색=리뷰, 초록=완료, 빨간=취소/긴급
- 플랫폼: 데스크톱 우선

## Core User Experience

### Defining Experience

**팀장의 핵심 루프:** 대시보드 열기 → 3초 안에 팀 상태 파악 → 할당/재할당/승인 → 닫기
**개발자의 핵심 루프:** CLI로 내 태스크 확인 → 코드 작업 → 상태 자동 갱신 + 대시보드로 팀 상황 확인

**Core Action:** 팀장과 개발자 모두 "현황 확인 + 조작"을 동등한 빈도로 수행. 대시보드는 읽기 전용이 아닌 조작 가능한 커맨드 센터.

### Platform Strategy

| 플랫폼 | 대상 사용자 | 우선순위 | 특성 |
|--------|-----------|---------|------|
| 웹 대시보드 | 팀장 + 개발자 | Primary | 데스크톱 우선, 마우스/키보드 중심 |
| CLI | 개발자 | Primary | 터미널 내 빠른 조회/변경 |
| 모바일 웹 | 팀장 | Future | Growth 단계 반응형 |

- 오프라인: Phase 3 (로컬 LLM)
- 키보드 단축키 지원 필수 (개발자 사용자)

### Effortless Interactions

1. **Zero-Input 상태 추론** — AI가 Git 활동에서 상태를 자동 추론. 개발자는 상태 업데이트를 "안 해도 된다". JIRA 대비 가장 강력한 차별점.
2. **자연어 → 태스크 분해** — "결제 시스템 구현해줘" 한 줄 → 에픽/스토리/태스크 .md 자동 생성. 백로그 작성 시간 제거.
3. **픽셀 오피스 한눈 파악** — 대시보드 열자마자 팀원들의 활동 상태가 시각적으로 즉시 전달. 통계 바를 읽을 필요 없이 "느낌"으로 파악.
4. **양방향 자동 싱크** — CLI에서 .md 수정 → 대시보드 자동 반영, 대시보드에서 드래그앤드롭 → Git 자동 커밋. 수동 동기화 불필요.

### Critical Success Moments

| 순간 | 설명 | 실패 시 영향 |
|------|------|------------|
| **First 3 Seconds** | 대시보드 열었을 때 팀 상태 즉시 파악 | "이건 JIRA랑 다를 게 없네" |
| **첫 AI 분해** | 자연어 입력 → 태스크 자동 생성 성공 | "AI가 쓸모없네" → 신뢰 상실 |
| **첫 자동 추론** | 코딩하다 보니 상태가 알아서 변경 | "결국 수동이네" → 핵심 가치 붕괴 |
| **첫 충돌 해결** | CLI↔대시보드 동시 수정 시 데이터 무손실 | "데이터 날아감" → 신뢰 붕괴 |
| **픽셀 오피스 발견** | 팀원이 캐릭터로 움직이는 것 확인 | 기능적 영향 없으나 차별점 약화 |

### Experience Principles

1. **Zero Friction** — 상태 업데이트, 태스크 생성, 동기화 등 모든 반복 작업에서 사용자 개입 최소화
2. **Glanceable Intelligence** — 3초 안에 팀 상태를 파악할 수 있는 정보 설계. 숫자보다 시각적 패턴 우선
3. **AI는 제안, 사람이 결정** — AI 추론 결과는 항상 "제안" 모드. 한 클릭으로 승인/거부. 자동 적용 없음
4. **Developer-Native** — CLI가 일급 시민. 대시보드는 CLI의 시각화 확장이지, CLI가 대시보드의 부속이 아님
5. **Fun & Alive** — 픽셀 오피스를 통해 팀 도구가 "살아있는" 느낌. 차갑고 기능적인 도구가 아닌 팀의 가상 공간

## Desired Emotional Response

### Primary Emotional Goals

| 감정 | 트리거 | 사용자 | 강도 |
|------|--------|--------|------|
| **해방감** | 상태 업데이트 안 해도 AI가 자동 처리 | 개발자 | 핵심 |
| **재미 + 소속감** | 픽셀 오피스에서 팀원 캐릭터 확인 | 전체 | 핵심 |
| **효율감** | AI가 PM 업무(분해, 추적, 보고) 자동화 | 팀장 | 핵심 |
| **안심** | 대시보드 열었을 때 팀 상태 한눈에 파악 | 팀장 | 높음 |
| **신뢰** | 문제 발생 시 시스템이 안전하게 처리 | 전체 | 높음 |

### Emotional Journey Mapping

| 단계 | 감정 | 설계 방향 |
|------|------|---------|
| 첫 발견 | 호기심 + "이건 뭐지?" | 픽셀 오피스 + 자연어 분해 데모로 즉각 흥미 유발 |
| 온보딩 | 자신감 | `sira init` 한 줄 → 즉시 시작. 복잡한 설정 없음 |
| 일상 사용 | 해방감 + 효율 | 수동 작업 제거. "안 해도 된다"의 연속 |
| 문제 발생 | 신뢰 | 충돌 시 양쪽 보존 + 명확한 안내. 데이터 손실 0 |
| 재방문 | 기대감 | 픽셀 오피스 변화, AI 추론 정확도 향상 체감 |

### Micro-Emotions

**강화할 감정:**
- **자신감 > 혼란** — 모든 UI 상태가 명확. "지금 뭘 해야 하지?"가 없음
- **신뢰 > 회의** — AI 추론에 항상 근거 제시 (커밋 해시, 파일 변경)
- **성취감 > 좌절** — 태스크 완료 시 픽셀 캐릭터 반응, 진행률 시각화
- **소속감 > 고립** — 팀원 활동이 실시간으로 보여 "함께 일하는" 느낌

**회피할 감정:**
- 감시당하는 느낌 → 개인 상세는 본인만 열람, 팀 뷰는 요약만
- 불안 → 충돌/오류 시 "데이터는 안전합니다" 명시적 안내
- 불신 → AI 제안에 항상 "왜?" 근거 표시

### Design Implications

| 감정 목표 | UX 설계 접근 |
|----------|------------|
| 해방감 | 자동 상태 추론 결과를 제안 뱃지로 표시. 승인은 한 클릭 |
| 재미 + 소속감 | 픽셀 오피스 캐릭터 애니메이션, 상태 변화 시 미묘한 반응 |
| 효율감 | 자연어 입력 → 결과 즉시 표시. 로딩 중에도 스트리밍 피드백 |
| 안심 | 글로벌 통계 바가 "정상/주의/위험" 색상으로 즉각 전달 |
| 신뢰 | 충돌 UI에 "양쪽 변경사항 보존됨" 명시. 복구 옵션 제공 |
| 자신감 | 모든 액션에 즉각 피드백. 빈 상태(empty state)에 가이드 |

### Emotional Design Principles

1. **Trust by Default** — 시스템은 항상 데이터를 보존한다. 사용자가 "괜찮을까?" 걱정할 일 없게
2. **Delight Through Discovery** — 픽셀 오피스의 숨은 반응, AI의 정확한 추론 등 "어, 이것도 되네?"의 작은 즐거움
3. **Liberation, Not Automation** — "AI가 대신 해준다"가 아니라 "내가 안 해도 된다"의 프레이밍. 사용자가 통제권을 유지
4. **Safety Net Visible** — 오류/충돌 상황에서 "시스템이 안전하게 처리 중" 메시지를 항상 표시. 불안 사전 차단

## UX Pattern Analysis & Inspiration

### Inspiring Products Analysis

**Linear (핵심 벤치마크)**
- 키보드 중심 조작 (Cmd+K 커맨드 팔레트)
- 극한의 로딩/전환 속도 (체감 즉시)
- 깔끔한 다크 테마 + 미니멀 UI
- SIRA 적용: 키보드 단축키 체계, 다크 테마 기본, 전환 속도 목표

**GitHub Projects**
- Git 저장소와 자연스러운 통합
- 이슈/PR과 태스크의 양방향 연결
- SIRA 적용: Git-native 데이터 모델의 UX 참고. 커밋/PR ↔ 태스크 연결 시각화

**Gather.town / 픽셀 오피스류**
- 픽셀아트 캐릭터로 팀원 존재감 시각화
- 실시간 상태를 공간적으로 표현
- SIRA 적용: 픽셀 오피스 뷰의 직접적 영감. 캐릭터 + 상태 뱃지 조합

**Notion**
- 동일 데이터를 테이블/칸반/타임라인 등 다양한 뷰로 전환
- 마크다운 기반 편집
- SIRA 적용: 멀티 뷰 패턴 (태스크 테이블 ↔ 간트 ↔ 프로젝트+멤버 뷰)

### Transferable UX Patterns

**Navigation Patterns:**
- **커맨드 팔레트 (Cmd+K)** — Linear 스타일. 모든 액션에 키보드로 접근. 개발자 사용자에게 필수
- **사이드바 네비게이션** — 프로젝트/뷰 전환. 접을 수 있는 사이드바로 화면 공간 확보

**Interaction Patterns:**
- **인라인 편집** — 테이블에서 셀 클릭 → 즉시 편집. 별도 페이지 이동 불필요
- **드래그앤드롭** — 칸반 보드에서 상태 변경 (목업에서 이미 계획됨)
- **멀티 뷰 전환** — 같은 데이터를 테이블/간트/프로젝트+멤버/픽셀오피스로 전환

**Visual Patterns:**
- **상태 색상 체계** — 노란색=진행중, 보라색=리뷰, 초록=완료, 빨간=취소/긴급 (목업 확정)
- **글로벌 통계 바** — 숫자 카드로 전체 현황 즉각 전달 (목업 확정)
- **아바타 + 이니셜** — 팀원 식별. 색상 코딩된 원형 아바타 (목업 확정)

### Anti-Patterns to Avoid

| Anti-Pattern | 문제점 | SIRA 대응 |
|-------------|--------|----------|
| 과도한 필드/폼 (JIRA식) | 태스크 생성에 10개+ 필드 → 포기 | 자연어 한 줄 입력 → AI 자동 분해 |
| 수동 상태 업데이트 강제 | "JIRA 업데이트 했어?" 반복 → 짜증 | AI 자동 추론 + 제안 모드 |
| 느린 페이지 로딩 | 3초+ 로딩 → 이탈 | Linear급 즉시 전환 목표 |
| 복잡한 권한 설정 UI | 관리자만 이해하는 설정 | 3역할 고정, 설정 최소화 |
| 알림 폭탄 | 모든 변경에 알림 → 무시 | 중요 이벤트만 선별 알림 |

### Design Inspiration Strategy

**Adopt (그대로 채택):**
- Linear의 키보드 단축키 체계 (Cmd+K 커맨드 팔레트)
- Linear의 다크 테마 + 빠른 전환 속도
- Notion의 멀티 뷰 전환 패턴

**Adapt (변형 적용):**
- Gather.town의 가상 공간 → SIRA 픽셀 오피스 (작업 상태 중심으로 단순화)
- GitHub Projects의 Git 연동 → SIRA의 양방향 싱크 (더 깊은 자동화)

**Innovate (SIRA 고유):**
- AI 상태 추론 제안 뱃지 UI — 기존 제품에 없는 패턴
- 사람+AI 동등 팀원 표시 — 기존 PM 도구에 없는 개념
- 자연어 → 태스크 자동 분해 인터랙션

## Design System Foundation

### Design System Choice

**Tailwind CSS + shadcn/ui** (Radix UI 기반)

### Rationale for Selection

| 기준 | shadcn/ui + Tailwind | Ant Design | 커스텀 |
|------|---------------------|------------|--------|
| Linear급 미니멀 | O (완전 제어) | X (과도한 스타일) | O (시간 소요) |
| 다크 테마 | O (Tailwind dark:) | O | O |
| 1인 개발 속도 | O (복사-붙여넣기) | O (풍부한 컴포넌트) | X (느림) |
| 번들 사이즈 | 작음 (트리쉐이킹) | 큼 | 최소 |
| 커스텀 컴포넌트 공존 | O (Tailwind 통일) | X (스타일 충돌) | O |
| 접근성(a11y) | O (Radix 내장) | O | 수동 |
| Next.js 호환 | 네이티브 | 호환 | 네이티브 |

**결정 이유:**
1. Linear급 미니멀 다크 테마를 자유롭게 구현 가능
2. 1인 개발에서 속도(기본 컴포넌트)와 유연성(완전 커스터마이징) 최적 밸런스
3. 픽셀 오피스(Canvas), 간트 차트 등 커스텀 컴포넌트와 Tailwind로 스타일 통일
4. Radix 기반 접근성 내장으로 별도 a11y 작업 최소화

### Implementation Approach

**Core Stack:**
- Tailwind CSS v4 — 유틸리티 CSS
- shadcn/ui — 기본 UI 컴포넌트 (Button, Dialog, Table, DropdownMenu, Command 등)
- Radix UI Primitives — headless 컴포넌트 (shadcn/ui 의존)
- Lucide Icons — 아이콘 세트 (shadcn/ui 기본)

**커스텀 컴포넌트 (SIRA 고유):**
- 픽셀 오피스 뷰 — HTML Canvas 또는 PixiJS
- 간트 차트 — Canvas 기반 커스텀 또는 경량 라이브러리
- AI 제안 뱃지 — shadcn Badge 확장
- 마크다운 렌더러/에디터 — react-markdown + 커스텀 에디터

### Customization Strategy

**Design Tokens (Tailwind 확장):**
- 색상 체계: 목업 확정 색상 → Tailwind custom colors
  - `status-progress: #EAB308` (노란색)
  - `status-review: #A855F7` (보라색)
  - `status-done: #22C55E` (초록색)
  - `status-critical: #EF4444` (빨간색)
  - `status-backlog: #6B7280` (회색)
- 다크 테마: 기본. 라이트 테마는 Growth 단계
- 타이포그래피: 시스템 폰트 (개발자 친화적, 한글 지원)
- 스페이싱: Tailwind 기본 4px 단위
- 라운딩: `rounded-lg` 기본 (목업 기준)

**컴포넌트 커스터마이징 우선순위:**
1. Table — 태스크 테이블 (프로젝트 그룹핑, 상태 배지, 인라인 편집)
2. Command — Cmd+K 커맨드 팔레트
3. Card — 멤버 활동 카드, 프로젝트 카드
4. Badge — 상태/우선순위/태그 뱃지
5. Dialog — 태스크 상세 모달

## Defining Core Experience

### Defining Experience

> **"한 줄 쓰면 프로젝트가 만들어진다"**

팀장이 "결제 시스템 구현해줘"라고 자연어를 입력하면, AI가 레포 컨텍스트를 분석하여 에픽/스토리/태스크 .md 파일을 자동 생성한다. 기존 PM 도구에서 백로그를 만드는 데 수 시간이 걸리던 작업이 한 줄로 끝난다.

**보조 Defining Experience:** "코딩만 하면 보드가 알아서 업데이트" — AI 상태 자동 추론은 일상 사용에서 핵심 가치를 매일 체감하게 하는 보조 경험.

### User Mental Model

**현재 사용자의 멘탈 모델:**
- JIRA: "에픽 만들고 → 스토리 쪼개고 → 태스크 생성하고 → 각각 필드 채우고..." (30분+)
- Linear: "이슈 만들고 → 라벨 붙이고 → 할당하고..." (10분)
- 사용자 기대: "아이디어를 말하면 끝" (10초)

**SIRA가 바꾸는 멘탈 모델:**
- Before: "백로그는 PM이 시간 들여 만드는 것"
- After: "아이디어를 말하면 AI가 구조화해 준다"

**혼란 가능 지점:**
- AI가 생성한 태스크 구조가 의도와 다를 때 → 편집 UI 필요
- 기존 태스크와 중복 생성될 때 → 중복 감지 + 알림
- "어디까지 분해해야 하지?" → AI가 적절한 깊이 제안

### Success Criteria

| 기준 | 목표 | 실패 |
|------|------|------|
| 입력 → 결과 시간 | 10초 이내 (스트리밍) | 30초+ 무응답 |
| 생성 품질 | 팀장이 70%+ 수정 없이 수락 | 50% 미만 수락률 |
| 컨텍스트 인식 | 기존 코드/태스크 참조 | 레포와 무관한 결과 |
| 편집 용이성 | 생성 후 인라인 수정 가능 | 전체 재생성 필요 |
| 학습 곡선 | 첫 사용에서 성공 | 설명서 필요 |

### Novel UX Patterns

**Novel (SIRA 고유):**
- **자연어 → 구조화된 .md 파일 생성** — 기존 PM 도구에 없는 인터랙션. 채팅이 아닌 "명령 → 결과물" 패턴
- **AI 제안 → 한 클릭 승인/거부** — AI 추론 결과를 뱃지로 표시, 사용자가 결정
- **스트리밍 생성** — 태스크가 하나씩 생성되는 과정을 실시간으로 보여줌

**Established (검증된 패턴 활용):**
- **커맨드 팔레트 (Cmd+K)** — Linear/VS Code에서 검증된 입력 방식
- **트리 구조 표시** — 에픽 > 스토리 > 태스크 계층을 접을 수 있는 트리로

**교육 전략:**
- 첫 사용 시 예시 프롬프트 제안 ("예: 사용자 인증 시스템 구현")
- 생성 중 "AI가 분석 중..." 단계별 피드백
- 빈 프로젝트에서 온보딩 가이드

### Experience Mechanics

**1. Initiation (시작):**
- CLI: `sira decompose "결제 시스템 구현해줘"`
- Dashboard: 상단 입력바 또는 Cmd+K → "decompose" → 자연어 입력
- 빈 프로젝트에서: "첫 백로그를 만들어 보세요" 가이드

**2. Interaction (진행):**
- 사용자가 자연어 입력 → Enter
- AI가 레포 구조 분석 (0.5초) → "분석 중..." 표시
- 스트리밍으로 에픽 → 스토리 → 태스크 순서로 생성
- 각 항목이 생성될 때마다 트리에 실시간 추가
- CLI: 스트리밍 텍스트 출력 / Dashboard: 트리 뷰 실시간 확장

**3. Feedback (피드백):**
- 생성 중: 진행률 + 현재 단계 표시 ("에픽 2/3 생성 중...")
- 각 태스크에 AI 신뢰도 표시 (높음/중간/낮음)
- 레포 참조 파일 목록 표시 ("관련 파일: payment.ts, cart.service.ts")
- 오류 시: "더 구체적으로 설명해 주세요" + 예시 제안

**4. Completion (완료):**
- "3개 에픽, 8개 스토리, 15개 태스크 생성 완료"
- 생성된 트리 구조 전체 미리보기
- [승인] [수정] [재생성] 3가지 액션 제공
- 승인 시: .md 파일 자동 생성 + Git commit
- 수정 시: 트리에서 인라인 편집 → 승인
