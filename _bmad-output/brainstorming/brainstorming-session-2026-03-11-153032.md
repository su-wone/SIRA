## Technique Execution Results

### Phase 1: Assumption Reversal (가정 뒤집기)

#### 가정 #1: "태스크는 계층 구조(에픽 → 스토리 → 태스크)로 관리해야 한다"

- **사용자 인사이트:** JIRA에서 익숙한 환경이라 편하다 — 관성적 결정임을 인정
- **결론:** 계층 구조 유지하되, 더 유연한 방식 탐색

**[아이디어 #1]: 하이브리드 뷰 — 저장은 플랫, 표시는 계층**
_Concept:_ .md 파일은 플랫하게 저장하되, AI가 메타데이터 기반으로 보는 사람에 따라 다른 뷰를 생성. 팀장은 에픽 뷰, 개발자는 "내 할 일" 뷰, PM은 진행률 뷰.
_Novelty:_ JIRA는 구조를 강제하지만, SIRA는 같은 데이터를 다른 렌즈로 보여주는 것.

**[아이디어 #2]: AI 자동 계층 진화**
_Concept:_ 초기에는 AI가 3단 계층으로 생성하지만, 프로젝트 진행 중 태스크 간 실제 관계를 학습해서 계층을 자동 재구성 제안.
_Novelty:_ 정적 계층이 아니라 살아 움직이는 구조.

#### 가정 #2: "팀장이 백로그를 생성하고 할당한다"

- **사용자 인사이트:** 양방향 가능하지만, 팀장 승인은 필수

**[아이디어 #3]: 승인 게이트가 있는 양방향 백로그**
_Concept:_ 팀원/AI가 태스크를 제안하면 `status: proposed` 상태의 .md 생성, 팀장 승인 시 `status: todo` 전환. "승인 큐" 관리.
_Novelty:_ 누구나 제안하되 흐름은 팀장이 통제. 마찰 없는 제안 프로세스.

**[아이디어 #4]: AI 코드 감시자 → 자동 태스크 제안**
_Concept:_ AI가 레포의 코드 변경, TODO 주석, 테스트 커버리지 하락 등을 모니터링해서 `status: ai-suggested` 태스크 자동 생성.
_Novelty:_ 사람이 인지하지 못한 기술 부채를 AI가 선제적으로 표면화.

#### 가정 #3: "태스크 데이터는 .md 파일에만 저장한다"

- **사용자 인사이트:** 인덱스 캐시 방식을 생각 못했는데 괜찮은 것 같다

**[아이디어 #5]: .md 파일 = Source of Truth + 자동 생성 인덱스**
_Concept:_ `.ai-collab/index.json`이 모든 .md의 frontmatter를 캐싱. .md 변경 시 Git hook/파일 워처가 인덱스 자동 갱신. 쿼리/필터는 인덱스에서, 상세 내용은 .md에서.
_Novelty:_ Git의 단순함 유지 + DB급 쿼리 성능. "MD를 DB처럼 쓴다"의 기술적 실현.

**[아이디어 #6]: Frontmatter 스키마 강제 + 마이그레이션**
_Concept:_ `.ai-collab/schema.yaml`로 frontmatter 필드 정의, 스키마 변경 시 AI가 기존 .md 파일 frontmatter 자동 마이그레이션.
_Novelty:_ DB의 migration 개념을 .md에 적용.

**[아이디어 #7]: CLI 쿼리 언어 — "md-query"**
_Concept:_ `ai-collab query "status:todo assignee:김철수 priority:high"` 명령으로 인덱스 즉시 필터링. `sort:priority`, `group:epic` 지원.
_Novelty:_ 터미널에서 JIRA 보드 대체. CLI-native 프로젝트 관리의 핵심 경험.

**[아이디어 #8]: AI 자연어 쿼리**
_Concept:_ `ai-collab ask "이번 주 마감인 백엔드 태스크 뭐 남았어?"` — AI가 자연어를 인덱스 쿼리로 변환.
_Novelty:_ 구조화 쿼리 + 자연어 동시 지원. AI-native 경험.

#### 가정 #4: "사람이 태스크 상태를 수동으로 업데이트한다"

**[아이디어 #9]: Git 활동 기반 자동 상태 추론**
_Concept:_ `related_files` 수정 시 → `in_progress`, PR 머지 → `review`, 팀장 승인 → `done`. 사람은 상태를 직접 안 바꿔도 됨.
_Novelty:_ "보드 업데이트 잊었어요" 문제 근본 해결. 프로젝트 상태 = 코드 활동의 자동 반영.

**[아이디어 #10]: AI 컨텍스트 인식 상태 추론**
_Concept:_ Claude Code 작업 대화를 AI가 분석해서 현재 작업 중인 태스크 자동 판별 + 상태 업데이트 + 예상 완료 시간 추정.
_Novelty:_ 작업 컨텍스트 자체가 프로젝트 관리 데이터. 관리 오버헤드 제로.

#### 가정 #5: "CLI가 MVP의 주요 인터페이스다"

**[아이디어 #11]: VS Code 확장 — .md 파일을 칸반 보드로 렌더링**
_Concept:_ `.ai-collab/` 폴더의 .md 파일들을 VS Code 사이드바에서 칸반 보드 표시. 드래그앤드롭으로 상태 변경 시 frontmatter 자동 수정.
_Novelty:_ 이미 열어놓은 도구 안에서 프로젝트 관리. 컨텍스트 스위칭 제로.

**[아이디어 #12]: GitHub/GitLab 렌더링 최적화**
_Concept:_ .md 파일이 GitHub에서 자연스럽게 읽히도록 뱃지, 체크리스트, 링크 자동 생성. 별도 도구 없이 GitHub이 대시보드.
_Novelty:_ 추가 설치 없이 GitHub 자체가 대시보드.

#### 가정 #6: "각 태스크는 한 명에게 할당된다"

**[아이디어 #13]: Human-AI 페어 할당**
_Concept:_ `assignee` + `ai_agent` 두 필드. 사람은 의사결정/리뷰, AI는 코드 생성/테스트. `ai_coverage: 80%` 메트릭 추적.
_Novelty:_ 관리 단위가 "사람+AI 팀의 작업"으로 진화.

**[아이디어 #14]: AI 난이도 자동 추정 + 자동 할당 제안**
_Concept:_ AI가 코드 복잡도 분석 + 팀원 과거 이력 기반 최적 할당 제안. 팀장은 승인/수정만.
_Novelty:_ 할당이 감이 아니라 데이터 기반 추천. 과부하 방지.

#### 가정 #7: "프로젝트 = 하나의 레포"

**[아이디어 #15]: 크로스 레포 태스크 관리**
_Concept:_ 하나의 에픽이 여러 레포에 걸쳐 하위 태스크 생성. 중앙 인덱스가 전체 통합.
_Novelty:_ 마이크로서비스 환경의 크로스 프로젝트 관리를 Git-native로.

#### 가정 #8: "태스크의 수명은 생성 → 완료"

**[아이디어 #16]: 반복 태스크 템플릿**
_Concept:_ `type: recurring` 태스크는 완료 시 자동으로 다음 스프린트에 새 인스턴스 생성.
_Novelty:_ JIRA 자동화 룰을 frontmatter 한 줄로 구현.

**[아이디어 #17]: 태스크 DNA — 완료 태스크에서 학습**
_Concept:_ 완료된 태스크의 실제 소요 시간, 커밋, 이슈를 분석해 유사 태스크에 인사이트 제공.
_Novelty:_ 프로젝트의 역사가 축적되는 지능형 관리 도구.

#### 가정 #9: "마크다운은 텍스트만 담는다"

**[아이디어 #18]: 실행 가능한 체크리스트**
_Concept:_ 체크박스 체크 시 연결된 테스트 자동 실행. `<!-- test: npm test -- payment.test.ts -->` 형식.
_Novelty:_ 마크다운 체크리스트가 실제 검증 도구.

**[아이디어 #19]: 인라인 AI 명령어**
_Concept:_ .md 안에 `<!-- ai: 이 태스크를 3개 하위 태스크로 분해해줘 -->` 삽입, AI가 감지 후 실행.
_Novelty:_ 마크다운 파일 자체가 AI와의 인터페이스.

### Phase 2: Cross-Pollination (도메인 간 교차 수분)

#### 도메인 1: 게임 — 퀘스트 시스템

**[아이디어 #20]: 퀘스트 트리 + 경험치 시스템**
_Concept:_ 태스크 = 퀘스트, 에픽 = 메인 퀘스트. 완료 시 XP 획득, 레벨업. 게이미피케이션 동기부여.
_Novelty:_ 프로젝트 관리에 게임 메카닉 도입.

**[아이디어 #21]: 스킬 트리 — 팀원 성장 추적**
_Concept:_ 완료 태스크 labels 기반 자동 스킬 맵. "backend Lv.5, payment Lv.3". AI 할당 시 스킬 매칭 활용.
_Novelty:_ 프로젝트 관리 도구가 팀원 성장 플랫폼 겸.

**[아이디어 #22]: 사이드 퀘스트 — 선택적 개선 작업**
_Concept:_ AI 발견 개선 사항을 `type: side-quest`로 생성. 여유 시간에 처리, 보너스 XP.
_Novelty:_ 기술 부채 관리를 의무가 아닌 보너스로 전환.

#### 도메인 2: 위키/노트 앱 — Obsidian/Notion

**[아이디어 #23]: 백링크 기반 태스크 연결**
_Concept:_ `[[payment-api-migration]]` 위키 링크로 양방향 참조. 그래프 뷰로 관계 시각화.
_Novelty:_ 계층 + 네트워크 구조 동시 지원. 숨겨진 의존성 발견.

**[아이디어 #24]: 데일리 노트 자동 생성**
_Concept:_ 매일 변경된 태스크, 커밋, 완료 항목 자동 기록. 스탠드업 대체.
_Novelty:_ 스탠드업 준비를 AI가 자동으로.

#### 도메인 3: 음악 프로덕션 — DAW

**[아이디어 #25]: 타임라인 뷰**
_Concept:_ 팀원 = 트랙, 태스크 = 클립. 병렬 작업과 의존성을 직관적으로 시각화.
_Novelty:_ 간트 차트의 재발명 — 더 직관적이고 인터랙티브.

**[아이디어 #26]: 믹싱 콘솔 — 프로젝트 밸런스 뷰**
_Concept:_ 영역별 투입 리소스와 우선순위 분포를 믹싱 채널처럼 표시. 불균형 시각화.
_Novelty:_ 리소스 배분을 직관적으로 느끼는 인터페이스.

#### 도메인 4: 소셜 미디어

**[아이디어 #27]: 프로젝트 피드**
_Concept:_ 모든 태스크 변경사항이 타임라인 피드로. 팀 활동 실시간 확인.
_Novelty:_ 프로젝트 관리의 소셜 미디어화 — 가시성 극대화.

**[아이디어 #28]: 리액션 기반 피드백**
_Concept:_ 태스크에 이모지 리액션. frontmatter에 `reactions` 저장. 비동기 피드백.
_Novelty:_ 비동기 커뮤니케이션 마찰 최소화.

#### 도메인 5: 생태계/자연

**[아이디어 #29]: 프로젝트 건강도 생태계 뷰**
_Concept:_ 건강한 영역 = 푸른 숲, 기술 부채 = 황무지, 활발한 개발 = 꽃. AI가 자동 산출.
_Novelty:_ 프로젝트 상태를 감정적으로 와닿게 시각화.

**[아이디어 #30]: 시즌 기반 개발 리듬**
_Concept:_ 봄=새 기능, 여름=스케일링, 가을=릴리즈, 겨울=리팩토링. 자연스러운 리듬.
_Novelty:_ 번아웃 방지 + 기술 부채 관리를 프로세스에 내장.

#### 도메인 6: 금융 — 포트폴리오 관리

**[아이디어 #31]: 리스크 포트폴리오 뷰**
_Concept:_ 태스크/에픽의 리스크를 금융 포트폴리오처럼 관리. 전체 스프린트 리스크 노출도 산출.
_Novelty:_ 프로젝트 리스크 관리에 금융 이론 적용.

**[아이디어 #32]: 기술 부채 이자율**
_Concept:_ 미뤄진 리팩토링에 이자율 부여. 방치 시 이자 누적. "이자율 15% — 2스프린트 후 작업량 2배".
_Novelty:_ 기술 부채를 정량적으로 체감.

### Phase 3: What If Scenarios (극한 시나리오)

#### 시나리오 1: "만약 AI가 팀의 80%를 차지한다면?"

**[아이디어 #33]: AI 에이전트 팀 관리**
_Concept:_ 사람 2명 + AI 8명 동시 관리. 각 AI에 전문 분야 부여. AI 작업 상태도 .md로 추적.
_Novelty:_ 프로젝트 관리 도구 → AI 오케스트레이션 플랫폼.

**[아이디어 #34]: AI 에이전트 성과 평가**
_Concept:_ AI의 코드 품질, 완료 속도, 버그율 자동 추적. 성과 낮으면 프롬프트 튜닝/모델 교체 제안.
_Novelty:_ AI 팀원에 대한 성과 관리 시스템.

#### 시나리오 2: "만약 인터넷이 없다면?"

**[아이디어 #35]: 완전 오프라인 모드**
_Concept:_ 로컬 LLM (Ollama) + Git만으로 동작. 온라인 복귀 시 자동 머지.
