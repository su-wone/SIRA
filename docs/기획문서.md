## 1. 프로젝트 개요

### 1.1 배경 및 필요성

AI 에이전트(Claude, GPT 등)가 개발팀의 실질적 팀원으로 활동하는 시대가 되었다. 그러나 기존 프로젝트 관리 도구(JIRA, Linear, Notion)는 "사람이 수동으로 상태를 관리하는" 패러다임에 머물러 있다.

**핵심 문제:**
- AI 에이전트의 작업을 사람과 동등하게 추적/관리하는 도구가 없음
- 개발자가 코드 작업 중 별도 도구로 전환하여 태스크를 업데이트해야 함
- 팀장이 사람과 AI의 전체 작업 현황을 한 곳에서 파악할 수 없음
- 태스크 상태가 실제 코드 진행과 동기화되지 않음

### 1.2 비전

> **"AI+Human 하이브리드 팀을 위한 작업 인텔리전스 플랫폼"**

SIRA는 사람과 AI 에이전트가 동등한 팀원으로 협업하며, Git 활동에서 자동으로 작업 상태를 추론하여 팀장에게 실시간 가시성을 제공한다.

### 1.3 핵심 차별점

| 기존 도구 | SIRA |
|----------|------|
| 사람이 수동으로 상태 업데이트 | Git 활동에서 AI가 자동 추론 |
| AI는 외부 연동 수준 | AI가 동등한 팀원으로 대시보드에 표시 |
| 전용 서버/DB 필요 | Git이 유일한 저장소 (DB 없음) |
| 웹 UI만 제공 | CLI + 웹 대시보드 양방향 싱크 |
| 텍스트 기반 상태 표시 | 픽셀 오피스로 팀 활동 시각화 |

## 2. 타겟 사용자

### 2.1 주요 페르소나

#### 팀장/PM — 민수 (32세)
- **역할:** 5인 개발팀 리더, 스프린트 관리
- **Pain Point:** JIRA에서 팀원들에게 상태 업데이트를 계속 요청해야 함
- **기대 가치:** 대시보드 하나로 팀 전체 (사람+AI) 현황 실시간 파악
- **주 인터페이스:** 웹 대시보드

#### 개발자 — 지영 (28세)
- **역할:** 풀스택 개발자, 매일 코드 작성
- **Pain Point:** IDE/터미널 ↔ JIRA 전환이 흐름을 끊음
- **기대 가치:** 터미널에서 태스크 조회/변경, 코드 커밋만으로 상태 자동 반영
- **주 인터페이스:** CLI

#### AI 에이전트 — Claude
- **역할:** 코드 작성, 리뷰, 태스크 분해 실행
- **동작:** `.sira/` 프로토콜을 따라 .md 파일 생성/수정 → Git push
- **주 인터페이스:** 프로그래매틱 (파일 시스템 + Git)

### 2.2 팀 규모

- **MVP:** 단일 팀, 2-10명 (사람 + AI 에이전트)
- **Growth:** 멀티 프로젝트, 외부 팀 5개, 태스크 5,000개

## 3. 핵심 기능

### 3.1 Git-Native 데이터 모델

모든 태스크는 YAML frontmatter가 포함된 `.md` 파일로 Git 레포에 저장된다.

```
tasks/
├── epics/       # 에픽 .md 파일
├── web/         # FE 태스크
├── server/      # BE 태스크
└── shared/      # 공통 태스크
```

- **인덱스 캐시:** `.sira/index.json` — frontmatter 캐싱으로 고속 쿼리
- **증분 업데이트:** 변경된 파일만 재인덱싱 (2초 이내)
- **자동 복구:** 인덱스 불일치 시 풀 리빌드 fallback

### 3.2 CLI (개발자용)

| 명령 | 기능 |
|------|------|
| `sira init` | 프로젝트 초기화 (.sira/, tasks/ 생성) |
| `sira query` | 태스크 조회 (필터/정렬/그룹핑) |
| `sira status` | 태스크 상태 변경 (RBAC 검증) |
| `sira assign` | 태스크 할당 |
| `sira decompose` | 자연어 → 태스크 자동 분해 (Claude Code 연동) |
| `sira ask` | 자연어 프로젝트 질의 |

### 3.3 웹 대시보드 (팀장용)

| 뷰 | 기능 |
|----|------|
| Dashboard | 글로벌 통계 바 + 픽셀 오피스 + 팀 현황 |
| Tasks | 태스크 테이블 (필터/정렬/인라인 편집) |
| Kanban | 칸반 보드 (드래그앤드롭 상태 변경) |
| Gantt | 간트 차트 (프로젝트별 타임라인) |
| Members | 멤버 목록 + 프로필 (사람/AI 동등) |
| Projects | 프로젝트별 진행률 비교 |

### 3.4 양방향 싱크

```
개발자 (CLI) → .md 수정 → git push
                                    ↘
                              GitHub Webhook → 대시보드 갱신 (SSE)
                                    ↗
팀장 (대시보드) → 드래그앤드롭 → .md 수정 → git push
```

- **필드 레벨 머지:** 서로 다른 필드 변경 시 자동 병합
- **충돌 해결:** 같은 필드 충돌 시 사용자 선택 요청, 양쪽 데이터 보존

### 3.5 AI 인텔리전스

- **자동 상태 추론:** 커밋 → `in_progress`, PR 생성 → `review`, PR 머지 → `done`
- **제안 모드:** AI 추론 결과를 뱃지로 표시, 팀장이 한 클릭 승인/거부
- **자연어 질의:** "이번 주 완료된 태스크는?", "팀에서 가장 바쁜 사람은?"

### 3.6 팀 & 권한 (RBAC)

| 역할 | 권한 |
|------|------|
| admin | 생성, 할당, 상태변경, 승인, 설정관리 |
| member | 본인 태스크 상태 변경 |
| agent | 할당된 태스크 상태 변경 |

### 3.7 픽셀 오피스

- Canvas 기반 RPG 스타일 가상 오피스
- 팀원(사람+AI)이 픽셀 캐릭터로 표시
- 캐릭터 상태가 실시간 작업 상태 반영 (작업중/대기/완료)
- SIRA 고유 차별점

## 5. 기술 아키텍처

### 5.1 모노레포 구조 (Turborepo)

```
@sira/core     — 스키마, 파서, 인덱서, 쿼리, RBAC, config, sync, inference
sira (CLI)     — init, query, status, assign, decompose, ask
web            — Next.js 16 대시보드
@sira/ui       — 공유 UI 컴포넌트
@sira/tsconfig — 공유 TypeScript 설정
```

### 5.2 의존 방향

```
apps/web  ──→  packages/core  ←──  apps/cli
```

web과 cli는 서로 직접 의존하지 않으며, core만 공유한다.

### 5.3 AI 연동 모델

SIRA 자체에 AI 파이프라인을 내장하지 않는다. 각 팀원의 로컬 Claude Code가 `.sira/` 프로토콜을 따라 .md 파일을 조작한다.

**장점:**
- 중앙 API 비용/Rate Limit 이슈 제거
- 각자 본인의 API 키 사용
- SIRA core가 AI 의존성 없이 단순
